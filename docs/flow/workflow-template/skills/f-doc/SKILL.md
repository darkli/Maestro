# 文档与模板开发 Skill

## 概述

本 Skill 提供 **5 阶段文档开发流水线**，专为文档/模板/prompt 类工作设计。核心差异化是**阶段 4 一致性扫描**——由专用 Agent 自动检测跨文件引用断裂、版本号不同步、README 未更新等文档特有问题。

**适用场景**：
- 工作流模板文件（SKILL.md、Agent .md）的新增或修改
- README、速查手册等文档的编写和更新
- prompt 文件（Agent 指令）的优化
- 跨多个文件的同步修改（如重命名某个 Agent 后需要更新所有引用处）
- 配置文件变更后需要同步更新文档

**不适用场景**：
- 功能代码开发 → 使用 `/f-dev`、`/f-quick-dev` 或 `/f-light-dev`
- Bug 修复 → 使用 `/f-bugfix`
- 纯设计方案 → 使用 `/f-design`

**与代码开发 Skill 的核心区别**：
- 无 TDD、编码、集成验证等代码专属阶段
- 新增一致性扫描阶段（7 类跨文件一致性检查）
- 审查维度聚焦文档质量（清晰度、完整性、一致性），而非代码质量（安全、性能）
- 阶段 1-3 由主协调器内联执行（不调用子代理），仅阶段 4-5 调用专用 Agent

## Workspace 工作记忆

### 目录结构

```
.claude/workspace/doc-YYYYMMDD-$NAME/
├── 00-input.md              # Skill 记录用户原始需求
├── 01-analysis.md           # 阶段 1 输出（需求分析）
├── 02-plan.md               # 阶段 2 输出（修改计划 + 一致性规则）
├── 03-changes.md            # 阶段 3 输出（变更清单摘要）
├── 04-consistency.md        # 阶段 4 输出（一致性扫描报告）
└── 05-review.md             # 阶段 5 输出（审查报告）
```

### 命名规范

```
doc-YYYYMMDD-短名称

示例：
  doc-20260223-skill-upgrade
  doc-20260225-readme-rewrite
  doc-20260301-agent-prompt-opt
```

### 核心规则

1. 每个 Agent 的 prompt 必须指定读哪些 Workspace 文件
2. 每个阶段完成后必须将产出摘要写入对应的 Workspace 文件
3. 文档变更写到项目正常位置，Workspace 文件存摘要和决策记录

## 流程总览

```
初始化              阶段 1         阶段 2         阶段 3          阶段 4            阶段 5
创建 Workspace ──→ 需求分析 ──→ 设计计划  ──→  编写实现  ──→  一致性扫描  ──→  文档审查
  │                  │            │              │               │                 │
  ▼                  ▼            ▼              ▼               ▼                 ▼
00-input.md       01-analysis  02-plan        03-changes     04-consistency     05-review
  │                  │            │              │               │                 │
                    内联         ⏸ 确认         内联            Agent             ⏸ 确认
                                                            (doc-consistency-    (doc-reviewer)
                                                              checker)
```

## 用户确认策略

**2 个确认点**：

| 确认点 | 位置 | 确认内容 | 为什么需要确认 |
|--------|------|----------|----------------|
| **设计计划确认** | 阶段 2 → 3 | 修改范围、文件清单、一致性规则是否正确 | 范围错了后续全部白费 |
| **审查确认** | 阶段 5 → 完成 | 一致性问题是否已修复、文档质量是否达标 | 用户决定质量标准 |

**确认时的用户选项**：

阶段 2 确认点：
- **确认通过** → 进入阶段 3
- **要求修改** → 重新分析设计
- **终止流程** → 保留 Workspace 已有产物

阶段 5 确认点：
- **确认通过** → 流程完成，输出总结
- **要求修复** → 返回阶段 3 修复后重新阶段 4-5（最多 1 轮）
- **部分接受** → 标记哪些必须修、哪些可后续处理

## 前置步骤：初始化 Workspace

### 脚本初始化

执行初始化脚本：

```bash
bash .claude/scripts/dev-init.sh --type=doc --name="$DOC_NAME" --input="$USER_INPUT" --skip-context
```

解析 JSON 输出：
- `workspace` → 赋值给 `$WS`，后续所有阶段使用此路径
- `warnings` → 非空时展示给用户
- `progress_updated` → 为 true 时告知用户已更新项目进度

注意：`--skip-context` 表示不生成 00-context.md，因为文档类工作的上下文由阶段 1 分析阶段直接从项目文件获取。

将 `$WS` 路径传递给后续每个阶段。

## 阶段 1：需求分析（主协调器内联）

**目标**：理解修改需求，识别涉及的文件和影响范围

**执行者**：主协调器（Skill 自身，不调用子代理）

**为什么内联**：文档需求通常简单明确，不需要子代理级别的分析能力

**执行步骤**：

1. 读取 `$WS/00-input.md` 了解用户需求
2. 根据需求类型定位相关文件：
   - 工作流模板修改 → 扫描 `docs/flow/workflow-template/` 目录结构
   - 项目文档修改 → 扫描项目文档目录
   - 已安装工作流修改 → 扫描 `.claude/` 目录结构
3. 识别所有受影响的文件（直接修改 + 需要同步更新的关联文件）
4. 列出变更类型（新增/修改/删除）

**输出** → 写入 `$WS/01-analysis.md`：

```markdown
# 需求分析

## 需求理解
[对用户需求的理解摘要]

## 涉及文件清单

### 直接修改
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| path/to/file | 新增/修改/删除 | 具体变更内容 |

### 关联同步（需要检查或更新）
| 文件 | 关联原因 | 预期操作 |
|------|----------|----------|
| path/to/file | 引用了目标文件 | 更新引用 |

## 影响范围评估
[对变更范围和风险的简要评估]
```

**检查点**：
```bash
git add "$WS/00-input.md" "$WS/01-analysis.md"
git commit -m "docs($DOC_NAME): 完成需求分析"
```

**阶段门禁**（自动流转，不暂停）：
- [ ] `$WS/01-analysis.md` 已创建
- [ ] 涉及文件清单已列出
- 自动进入阶段 2

---

## 阶段 2：设计计划（主协调器内联）

**目标**：制定详细修改计划，并定义本次变更需要检查的一致性规则

**执行者**：主协调器（Skill 自身）

**执行步骤**：

1. 读取 `$WS/01-analysis.md` 了解涉及文件和影响范围
2. 读取每个涉及文件的当前内容，理解现有结构
3. 制定每个文件的具体修改方案
4. **定义一致性规则**：根据本次变更的内容，确定阶段 4 需要检查的一致性类别和具体检查点

**一致性规则定义**：

从以下 7 类一致性检查中，选取与本次变更相关的类别，并定义具体检查点：

| 类别 ID | 类别名称 | 何时需要 |
|---------|----------|----------|
| C1 | 跨文件引用完整性 | 任何涉及文件名、路径变更的修改 |
| C2 | 版本号同步 | 涉及模板文件版本变更时 |
| C3 | 模板占位符残留 | 涉及模板生成逻辑修改时 |
| C4 | 命名约束遵守 | 涉及 Skill/Agent/Workspace 文件名变更时 |
| C5 | README/速查手册同步 | 任何新增/删除/重命名文件或功能变更 |
| C6 | 交叉引用名称同步 | SKILL 中引用 Agent 文件名或 Workspace 文件名变更时 |
| C7 | 模板与安装副本一致 | 修改 template/ 下文件且有对应的 .claude/ 安装副本时 |

**输出** → 写入 `$WS/02-plan.md`：

```markdown
# 修改计划

## 修改方案

### 文件 1: path/to/file
**变更类型**: 新增/修改/删除
**具体内容**:
[详细描述要做的修改]

### 文件 2: path/to/file
...

## 一致性规则定义

本次变更需要检查以下一致性规则：

### 适用的检查类别
| 类别 | 适用原因 | 具体检查点 |
|------|----------|-----------|
| C1 跨文件引用 | [为什么需要] | [具体要检查什么] |
| C5 README 同步 | [为什么需要] | [具体要检查什么] |
...

### 不适用的检查类别
| 类别 | 不适用原因 |
|------|-----------|
| C3 占位符残留 | 本次变更不涉及模板生成逻辑 |
...

## 修改顺序
1. [建议的文件修改顺序和原因]

## 风险点
- [潜在问题或需要特别注意的地方]
```

**检查点**：
```bash
git add "$WS/02-plan.md"
git commit -m "docs($DOC_NAME): 完成修改计划"
```

**⏸ 阶段门禁**（需用户确认）：
- [ ] `$WS/02-plan.md` 已创建
- [ ] 修改方案已制定
- [ ] 一致性规则已定义
- [ ] 用户确认方案

向用户展示 02-plan.md 的核心摘要，询问：
- 确认通过 / 要求修改 / 终止

---

## 阶段 3：编写实现（主协调器内联）

**目标**：按计划执行文档修改

**执行者**：主协调器（Skill 自身）

**为什么内联**：文档编写不需要专门的子代理，主协调器有足够的能力完成

**执行步骤**：

1. 读取 `$WS/02-plan.md` 了解修改方案和修改顺序
2. 按修改顺序逐个文件执行变更
3. 对每个文件：
   - 读取当前内容
   - 执行修改（新增内容/修改内容/删除内容）
   - 确认修改正确后写入
4. 记录所有实际执行的变更

**编写规范**：
- 遵循被修改文件的现有格式和风格
- Markdown 文件遵循一致的标题层级、列表格式
- 模板文件中的占位标记保持一致的格式
- frontmatter 格式严格遵循 YAML 规范
- 代码块中的缩进与语言规范一致

**输出** → 写入 `$WS/03-changes.md`：

```markdown
# 变更清单

## 变更摘要
| 文件 | 变更类型 | 状态 |
|------|----------|------|
| path/to/file | 新增/修改/删除 | 已完成 |

## 变更详情

### 文件 1: path/to/file
**变更类型**: 修改
**关键变更**:
- [变更点 1 摘要]
- [变更点 2 摘要]

### 文件 2: path/to/file
...

## 下游摘要

### 变更文件路径列表（用于 git add）
[每行一个路径，可直接用于构建 git add 命令]
path/to/file1
path/to/file2

### 一致性扫描范围
[需要进行一致性扫描的目录和文件范围]

### 适用的一致性规则
[从 02-plan.md 继承的一致性规则定义]
```

**检查点**：
从 `$WS/03-changes.md` 的"变更文件路径列表"读取路径，构建 git add 命令：
```bash
git add "$WS/03-changes.md" [从下游摘要读取的变更文件路径]
git commit -m "docs($DOC_NAME): 完成文档编写"
```

**阶段门禁**（自动流转，不暂停）：
- [ ] 所有计划内的文件变更已执行
- [ ] `$WS/03-changes.md` 已创建
- 自动进入阶段 4

---

## 阶段 4：一致性扫描

**目标**：自动检测跨文件一致性问题

**子代理**：Doc Consistency Checker（`doc-consistency-checker.md`，Sonnet）

**核心差异化**：这是 f-doc 区别于代码开发 Skill 的关键阶段。文档/模板工作中最常见的质量问题不是逻辑 Bug，而是跨文件引用断裂、版本号漏更新、README 未同步等一致性问题。

**子代理输入**：
- 读取 `$WS/02-plan.md`（获取一致性规则定义——知道要检查什么）
- 读取 `$WS/03-changes.md`（获取变更文件清单和扫描范围——知道要扫描哪里）

**执行步骤**：参见 `doc-consistency-checker.md` Agent 文件中的详细检查流程。

**子代理输出** → 写入 `$WS/04-consistency.md`

**一致性问题处理**：

```
无问题发现 → 自动进入阶段 5
发现问题 →
  CRITICAL/HIGH 级别 → 子代理自动修复（有修复权限）→ 修复后重新扫描确认（最多 1 轮）
  MEDIUM 级别 → 记录在报告中，由阶段 5 审查员评估
  LOW/INFO 级别 → 仅记录，不阻塞
```

**检查点**（如有修复）：
```bash
git add "$WS/04-consistency.md" [修复涉及的文件]
git commit -m "docs($DOC_NAME): 修复一致性问题"
```

**阶段门禁**（自动流转，不暂停）：
- [ ] `$WS/04-consistency.md` 已创建
- [ ] CRITICAL/HIGH 问题已修复或无此类问题
- 自动进入阶段 5

---

## 阶段 5：文档审查

**目标**：从文档质量和完整性角度审查变更

**子代理**：Doc Reviewer（`doc-reviewer.md`，Sonnet）

**子代理输入**：
- 读取 `$WS/02-plan.md`（修改方案——核对实现是否符合计划）
- 读取 `$WS/03-changes.md` 的"下游摘要"节（变更文件路径列表——只审查这些文件）
- 主协调器预读 `$WS/04-consistency.md` 的"下游摘要"节，将整体结论和残留问题内联传入 prompt（减少 token 开销）

**执行步骤**：参见 `doc-reviewer.md` Agent 文件中的详细审查流程。

**审查结果处理**：
```
APPROVE → 流程完成
REQUEST CHANGES → 返回阶段 3 修复 → 重新阶段 4-5（最多 1 轮）
APPROVE WITH COMMENTS → 记录建议，流程完成
```

**子代理输出** → 写入 `$WS/05-review.md`

**检查点**（如有修复）：
```bash
git add "$WS/05-review.md" [修复涉及的文件]
git commit -m "docs($DOC_NAME): 修复审查问题"
```

**⏸ 阶段门禁**（需用户确认）：
向用户展示 `$WS/05-review.md` 摘要（含一致性扫描结果总结）。

用户选项：
- **确认通过** → 流程完成，输出总结
- **要求修复** → 返回阶段 3 修复后重新审查
- **部分接受** → 标记必须修和可后续处理

**更新项目进度**（如属于多阶段项目）：
```
如果 .claude/workspace/_progress-*.md 中记录了本任务，更新对应子任务状态为"已完成"，
并记录 Workspace 路径和完成时间。如果所有子任务均已完成，标记项目整体为"已完成"。
```

## 检查点与恢复

### 检查点策略

```
commit 1: docs($DOC_NAME): 完成需求分析          → $WS/00 + 01
commit 2: docs($DOC_NAME): 完成修改计划          → $WS/02
commit 3: docs($DOC_NAME): 完成文档编写          → $WS/03 + 变更文件
commit 4: docs($DOC_NAME): 修复一致性问题（如有）→ $WS/04 + 修复文件
commit 5: docs($DOC_NAME): 修复审查问题（如有）  → $WS/05 + 修复文件
```

### 从指定阶段恢复

查找 `.claude/workspace/doc-*-$NAME/`，检查前序 Workspace 文件是否存在，从对应阶段继续。

## 注意事项

1. **阶段 1-3 内联执行**：主协调器直接执行，不调用子代理。这是因为文档分析和编写不需要 Agent 级别的专业能力，内联执行可以减少 token 开销和上下文切换。
2. **一致性扫描是核心价值**：阶段 4 是 f-doc 的核心差异化阶段，不可跳过。即使变更看似简单，也应执行一致性扫描。
3. **一致性规则是动态的**：阶段 2 根据本次具体变更内容定义需要检查的一致性类别，而非每次都检查全部 7 类。这避免了无关检查的干扰。
4. **修复权限**：阶段 4 的一致性检查 Agent 拥有修复权限（Read + Write + Edit + Bash + Grep + Glob），可以直接修复 CRITICAL/HIGH 问题。这与 code-reviewer 的只读策略不同，因为文档一致性问题（如版本号同步、引用更新）通常有明确的修复方式，不需要人工决策。
5. **与代码审查的区别**：阶段 5 的文档审查关注清晰度、完整性、格式一致性等维度，不包含安全审计和性能分析。
