# 文档与模板开发 Skill

本 Skill 是一份流程指引——定义文档/模板类工作应该经过的阶段和质量门禁，由 Claude 在主对话中直接执行。核心差异化是**一致性检查**——确保跨文件引用完整、版本号同步、README 更新等文档特有的质量问题。

## 入口逻辑

按以下优先级确定需求来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-doc` 后提供了文档任务描述 → 直接使用
2. **对话上下文**：当前对话中已讨论过文档修改需求 → 提取要点，向用户确认理解是否准确
3. **未完成任务**：`.claude/workspace/` 中存在最近 7 天内的未完成 `doc-*` 目录（缺少 02-summary.md）→ 询问用户是否继续
4. **无上下文**：以上都没有 → 请用户提供任务名称和描述

确定需求后，创建 Workspace 目录：

```bash
mkdir -p .claude/workspace/doc-$(date +%Y%m%d)-$NAME
```

## Workspace 归档

Workspace 是产物归档，不是上下文传递管道。

```
.claude/workspace/doc-YYYYMMDD-$NAME/
├── 01-plan.md        # 修改计划 + 一致性规则
└── 02-summary.md     # 变更总结
```

各文件内容指引：
- **01-plan.md**：需求理解、涉及文件清单、修改方案、适用的一致性规则、修改顺序
- **02-summary.md**：变更文件清单、一致性检查结果、审查结论

## 流程

### 阶段 1：分析与计划

1. **理解需求**
   - 从确定的需求来源提取任务目标
   - 如有不清晰之处，向用户澄清

2. **定位文件**
   - 根据需求类型扫描相关目录
   - 识别所有受影响的文件（直接修改 + 需要同步更新的关联文件）

3. **制定方案**
   - 每个文件的具体修改内容
   - 修改顺序和依赖关系

4. **定义一致性规则**
   - 根据本次变更内容，从下方一致性检查清单中选取适用的类别

5. **归档**：写入 `$WS/01-plan.md`

**⏸ 确认点**：向用户展示修改计划和范围，获得确认
- 确认通过 → 进入阶段 2
- 要求修改 → 调整后重新确认
- 终止 → 保留已有产物

### 阶段 2：编写实现

1. **执行修改**
   - 按计划逐个文件执行变更
   - 遵循被修改文件的现有格式和风格
   - Markdown 标题层级、列表格式保持一致
   - frontmatter 严格遵循 YAML 规范

### 阶段 3：一致性检查与审查

1. **一致性扫描**
   - 按阶段 1 定义的规则逐项检查
   - 参照下方一致性检查清单执行
   - 发现问题直接修复

2. **文档审查**
   - 清晰度：描述是否易懂、无歧义
   - 完整性：是否有遗漏的内容
   - 格式一致性：风格是否与项目现有文档统一
   - 计划符合度：实际变更是否与 `01-plan.md` 一致

3. **归档**：写入 `$WS/02-summary.md`（变更清单、一致性检查结果、审查结论）

**⏸ 确认点**：向用户展示变更总结和检查结果
- 确认完成 → 流程结束
- 要求修复 → 修复后重新检查

## 一致性检查清单

根据变更内容选取适用的类别：

| ID | 类别 | 何时适用 | 检查内容 |
|----|------|----------|----------|
| C1 | 跨文件引用完整性 | 涉及文件名、路径变更 | 所有引用该路径的文件是否已更新 |
| C2 | 版本号同步 | 涉及模板文件版本变更 | frontmatter version 是否一致 |
| C3 | 模板占位符残留 | 涉及模板生成逻辑修改 | 是否有未替换的 $PLACEHOLDER |
| C4 | 命名约束遵守 | 涉及 Skill/Workspace 名变更 | 是否符合项目命名规范 |
| C5 | README/速查手册同步 | 新增/删除/重命名文件或功能 | README 和文档索引是否已更新 |
| C6 | 交叉引用名称同步 | SKILL 引用其他 Skill 或 Workspace 名变更 | 引用名称是否与实际文件名匹配 |
| C7 | 模板与安装副本一致 | 修改 template/ 下文件 | .claude/ 安装副本是否需要同步 |

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：
1. 扫描 `.claude/workspace/doc-*` 找最近的未完成目录
2. 读取已有的 01/02 文件推断进度
3. 向用户确认后从中断点继续

**多阶段项目**：
- 如 `_progress-*.md` 中记录了本任务，流程完成后更新对应子任务状态

## 质量门禁

- 一致性扫描发现的 Critical/High 问题必须修复后才能进入确认
- 所有计划内的文件变更必须完成
- 跨文件引用必须完整（无断裂引用）

## 适用场景

- 工作流模板文件（SKILL.md）的新增或修改
- README、速查手册等文档的编写和更新
- 跨多个文件的同步修改（如重命名某个 Skill 后更新所有引用处）
- 配置文件变更后同步更新文档

不适合：
- 功能代码开发 → 用 `/f-dev`
- Bug 修复 → 用 `/f-bugfix`
- 纯设计方案 → 用 `/f-design`
