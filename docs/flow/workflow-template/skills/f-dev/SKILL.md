# 功能开发 Skill

本 Skill 是一份流程指引——定义功能开发应该经过的阶段和质量门禁，由 Claude 在主对话中直接执行，充分利用对话上下文的连续性。

## 入口逻辑

按以下优先级确定需求来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-dev` 后提供了功能描述 → 直接使用
2. **对话上下文**：当前对话中已讨论过需求或设计方案 → 提取要点，向用户确认理解是否准确
3. **设计上游**：`.claude/workspace/` 中存在 `design-*` 目录 → 列出最近 7 天内的候选（展示目录名和创建日期），询问用户是否基于某个设计继续
4. **未完成任务**：`.claude/workspace/` 中存在最近 7 天内的未完成 `feature-*` 目录（缺少 03-summary.md）→ 询问用户是否继续
5. **无上下文**：以上都没有 → 请用户提供功能名称（短名称，用于目录命名）和功能描述

确定需求后，创建 Workspace 目录：

```bash
mkdir -p .claude/workspace/feature-$(date +%Y%m%d)-$NAME
```

## Workspace 归档

Workspace 是产物归档，不是上下文传递管道。

```
.claude/workspace/feature-YYYYMMDD-$NAME/
├── 01-requirements.md    # 需求分析
├── 02-design.md          # 设计方案
└── 03-summary.md         # 开发总结
```

各文件内容指引：
- **01-requirements.md**：功能概述、核心功能点与验收标准、边界情况、影响范围
- **02-design.md**：架构方案、接口设计、数据模型变更、实现步骤（如多模块）
- **03-summary.md**：变更文件清单、测试结果、注意事项

## 流程

### 阶段 1：需求与设计

1. **理解需求**
   - 从确定的需求来源提取功能点、约束条件、验收标准
   - 如有不清晰之处，向用户澄清
   - 分析对现有模块的影响范围

2. **阅读代码**
   - 阅读相关源代码，理解现有架构和模式
   - 识别需要修改的文件和模块

3. **设计方案**
   - 模块划分与职责
   - API / 接口设计（如涉及）
   - 数据模型变更（如涉及）
   - 如涉及多个模块，列出实现步骤和顺序

4. **归档**：写入 `$WS/01-requirements.md` 和 `$WS/02-design.md`

**⏸ 确认点**：向用户展示设计方案要点，获得确认
- 确认通过 → 进入阶段 2
- 要求修改 → 调整方案后重新确认
- 终止 → 保留已有产物

### 阶段 2：编码实现

1. **实现代码**
   - 按设计方案实现（多模块按步骤逐个实现）
   - 遵循 CLAUDE.md 中的编码规范
   - 如 CLAUDE.md 配置了 i18n → 更新所有 locale 文件
   - 如 CLAUDE.md 配置了 design-system → 使用项目 design token，确保暗色模式兼容

2. **编写测试**（如 CLAUDE.md 配置了 testing）
   - 为新功能编写测试
   - 运行测试套件，修复失败的测试

### 阶段 3：审查与验证

1. **自查变更**（逐项检查，发现问题直接修复，确认点展示修复后的最终状态）

   **安全性**
   - 用户输入是否经过验证/转义（防注入、XSS）
   - 敏感信息是否暴露（API Key、密码、内部路径）
   - 权限检查是否完备（如涉及鉴权）

   **类型安全**（如 CLAUDE.md 配置了 static-types）
   - 类型是否完备（无 any 逃逸、接口定义与数据结构一致）
   - 可空值是否正确处理

   **边界与错误处理**
   - 空值、空数组、空字符串的处理
   - 网络错误、超时、并发竞态
   - 异常输入（超长、特殊字符、非预期类型）

   **性能**
   - 是否有不必要的重复计算或重渲染（如适用）
   - 数据查询是否有 N+1 问题（如涉及数据库）
   - 大数据量场景是否考虑（如适用）

   **可维护性**
   - 命名是否清晰（函数名体现意图，变量名无歧义）
   - 是否有可消除的重复代码
   - 函数/组件是否职责单一

   **项目规范一致性**
   - 代码风格与 CLAUDE.md Code Style 一致
   - 新增文件放在正确的目录
   - 遵循了 Adding New Features 的步骤

2. **运行验证**
   - 构建是否通过
   - 完整测试套件是否通过
   - i18n 完整性（如适用）

3. **归档**：写入 `$WS/03-summary.md`（变更摘要、涉及文件清单、测试结果）

**⏸ 确认点**：向用户展示变更总结
- 确认完成 → 流程结束
- 要求修改 → 修改后重新审查

## 复杂度自适应

不需要用户选择模式，Claude 根据任务自行调整：

- **简单功能**（单文件修改、UI 微调）：阶段 1 简化为几句话的方案确认，快速编码
- **中等功能**（跨几个文件、新增 API）：完整三阶段
- **复杂功能**（跨多模块、架构变更）：阶段 1 深入设计（架构图、接口定义），阶段 2 分步实现，阶段 3 细致审查

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：
1. 扫描 `.claude/workspace/feature-*` 找最近的未完成目录
2. 读取已有的 01/02/03 文件推断进度
3. 向用户确认后从中断点继续

**多阶段项目**：
- 如 `_progress-*.md` 中记录了本任务，流程完成后更新对应子任务状态
- 用户说"继续做"时，检查 progress 文件确定下一步

## 质量门禁

- 自查发现 Critical/High 问题必须修复后才能进入确认
- 测试必须通过（如项目配置了测试框架）
- i18n 所有 locale 文件必须同步更新（如项目配置了 i18n）
