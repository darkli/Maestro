# 需求分析与系统设计 Skill

本 Skill 是一份流程指引——只做需求分析和系统设计，不写代码。产出可直接衔接 `/f-dev` 进入开发。

## 入口逻辑

按以下优先级确定需求来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-design` 后提供了功能描述 → 直接使用
2. **对话上下文**：当前对话中已讨论过需求或想法 → 提取要点，向用户确认理解是否准确
3. **产品上游**：`.claude/workspace/` 中存在 `product-*` 目录 → 列出最近 7 天内的候选，询问用户是否基于某个产品设计继续
4. **无上下文**：以上都没有 → 请用户提供功能名称和描述

确定需求后，创建 Workspace 目录：

```bash
mkdir -p .claude/workspace/design-$(date +%Y%m%d)-$NAME
```

## Workspace 归档

Workspace 是产物归档，不是上下文传递管道。

```
.claude/workspace/design-YYYYMMDD-$NAME/
├── 01-requirements.md    # 需求文档
└── 02-design.md          # 设计方案
```

各文件内容指引：
- **01-requirements.md**：功能概述、核心功能点与验收标准、边界情况、非功能需求、约束条件、影响范围、风险评估
- **02-design.md**：架构概览（含 Mermaid 图）、模块划分、API 接口定义、数据模型、关键设计决策、文件结构、需求覆盖检查

## 流程

### 阶段 1：需求分析

1. **理解需求**
   - 从确定的需求来源提取功能点、约束条件
   - 如有不清晰之处，向用户澄清
   - 识别核心功能点、边界情况
   - 定义可测试的验收标准

2. **评估影响**
   - 分析对现有模块的影响范围
   - 识别需要修改的文件和模块

3. **归档**：写入 `$WS/01-requirements.md`

**⏸ 确认点**：向用户展示需求理解和验收标准，获得确认
- 确认通过 → 进入阶段 2
- 要求修改 → 调整后重新确认
- 终止 → 保留已有产物

### 阶段 2：系统设计

1. **阅读代码**
   - 阅读相关源代码，理解现有架构和模式
   - 理解项目的分层结构和数据流

2. **设计架构**
   - 模块划分与职责
   - API / 接口定义（如涉及）
   - 数据模型变更（如涉及）
   - 架构图（Mermaid）
   - 需求覆盖检查：逐条对照 `01-requirements.md` 的功能点

3. **归档**：写入 `$WS/02-design.md`

**⏸ 确认点**：向用户展示设计方案，获得确认
- 确认通过 → 流程完成
- 要求修改 → 调整后重新确认

## 完成与后续衔接

设计完成后，检查设计方案是否建议拆分为多个子任务：

**如果需要拆分**：创建项目进度文件 `.claude/workspace/_progress-$NAME.md`（格式遵循 CLAUDE.md 中 Multi-Phase Project Tracking 部分），告知用户可通过"继续做"恢复。

**后续使用**：`/f-dev` 会自动检测到 `design-*` 目录并提供使用选项。

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：
1. 扫描 `.claude/workspace/design-*` 找最近的未完成目录
2. 读取已有的 01/02 文件推断进度
3. 向用户确认后从中断点继续

**多阶段项目**：
- 如 `_progress-*.md` 中记录了本任务，流程完成后更新对应子任务状态
- 用户说"继续做"时，检查 progress 文件确定下一步

## 适用场景

- 前期方案规划（如"评估是否应该加用户角色系统"）
- 技术选型讨论（如"分析 WebSocket vs SSE 的方案"）
- 可行性分析（如"评估数据库迁移方案的影响"）
- 大功能前的方案设计（先设计再决定是否开发）
- 重构方案设计（如"设计前端状态管理的重构方案"）

不适合：
- 需要写代码的任务 → 用 `/f-dev`
- Bug 修复 → 用 `/f-bugfix`
- 代码分析讨论 → 直接与 Claude 对话
