# Bug 修复 Skill

本 Skill 是一份流程指引——定义 Bug 修复应该经过的阶段和质量门禁，由 Claude 在主对话中直接执行，充分利用对话上下文的连续性。

## 入口逻辑

按以下优先级确定 Bug 来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-bugfix` 后提供了 Bug 描述 → 直接使用
2. **对话上下文**：当前对话中已讨论过 Bug 现象或错误信息 → 提取要点，向用户确认理解是否准确
3. **未完成任务**：`.claude/workspace/` 中存在最近 7 天内的未完成 `bugfix-*` 目录（缺少 02-summary.md）→ 询问用户是否继续
4. **无上下文**：以上都没有 → 请用户提供 Bug 名称（短名称，用于目录命名）和 Bug 描述

确定 Bug 后，创建 Workspace 目录：

```bash
mkdir -p .claude/workspace/bugfix-$(date +%Y%m%d)-$NAME
```

## Workspace 归档

Workspace 是产物归档，不是上下文传递管道。

```
.claude/workspace/bugfix-YYYYMMDD-$NAME/
├── 01-diagnosis.md    # 根因分析 + 修复方案
└── 02-summary.md      # 修复总结
```

各文件内容指引：
- **01-diagnosis.md**：Bug 描述、复现路径、根因分析、影响范围、修复方案
- **02-summary.md**：变更文件清单、回归测试结果、注意事项

## 流程

### 阶段 1：诊断分析

1. **理解 Bug**
   - 从确定的 Bug 来源提取错误现象、触发条件
   - 如有不清晰之处，向用户澄清

2. **定位根因**
   - 阅读相关源代码，追踪数据流和控制流
   - 分析复现路径：什么操作序列触发了这个 Bug
   - 定位根因：问题出在哪个文件的哪个逻辑

3. **设计修复方案**
   - 评估影响范围：这个 Bug 还会影响哪些功能
   - 制定修复策略：怎么改、改哪些文件
   - 评估回归风险

4. **归档**：写入 `$WS/01-diagnosis.md`

**⏸ 确认点**：向用户展示根因分析和修复方案，获得确认
- 确认通过 → 进入阶段 2
- 补充信息 → 提供更多 Bug 上下文，重新诊断
- 调整方案 → 指定不同的修复方向
- 终止 → 保留已有产物

### 阶段 2：修复实现

1. **实现修复**
   - 按修复方案实现
   - 遵循 CLAUDE.md 中的编码规范
   - 确保修复的是根因，而非掩盖症状
   - 如 CLAUDE.md 配置了 i18n 且修复涉及 UI 文本 → 更新所有 locale 文件

2. **编写回归测试**（如 CLAUDE.md 配置了 testing）
   - 覆盖原始 Bug 场景
   - 覆盖相关边界情况
   - 运行测试套件，确保修复有效且不破坏其他功能

### 阶段 3：审查与验证

1. **自查变更**（逐项检查，发现问题直接修复，确认点展示修复后的最终状态）

   **根因验证**
   - 修复的是根因还是只掩盖了症状
   - 是否有其他代码路径存在相同的根因（同类 Bug 排查）

   **影响范围**
   - 修复是否引入新的边界情况
   - 修复范围是否超出必要（避免过度修改）
   - 对现有功能的回归风险

   **回归测试完备性**
   - 测试是否覆盖了原始 Bug 的精确触发条件
   - 测试是否覆盖了同类的相关边界情况
   - 测试断言是否足够精确（不只是 "no error"，而是验证正确行为）

   **项目规范一致性**
   - 代码风格与现有代码匹配
   - i18n 文本更新完整（如涉及 UI 文本修改）

2. **运行验证**
   - 构建是否通过
   - 完整测试套件是否通过

3. **归档**：写入 `$WS/02-summary.md`（修复摘要、涉及文件清单、测试结果）

**⏸ 确认点**：向用户展示修复总结
- 确认完成 → 流程结束
- 要求修改 → 修改后重新审查

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：
1. 扫描 `.claude/workspace/bugfix-*` 找最近的未完成目录
2. 读取已有的 01/02 文件推断进度
3. 向用户确认后从中断点继续

**多阶段项目**：
- 如 `_progress-*.md` 中记录了本任务，流程完成后更新对应子任务状态
- 用户说"继续做"时，检查 progress 文件确定下一步

## 质量门禁

- 自查发现 Critical/High 问题必须修复后才能进入确认
- 测试必须通过（如项目配置了测试框架）
- 回归测试必须覆盖原始 Bug 场景
