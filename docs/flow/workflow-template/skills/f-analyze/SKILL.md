# 深度问题分析 Skill

本 Skill 是一份流程指引——对 Bug 或异常行为进行深度调查，输出根因分析和修复建议。**全程不修改主分支代码、不产生 git commit/push**，产出仅为 Workspace 中的分析文档。

## 入口逻辑

按以下优先级确定问题来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-analyze` 后提供了问题描述 → 直接使用
2. **对话上下文**：当前对话中已讨论过错误现象或异常行为 → 提取要点，向用户确认理解是否准确
3. **未完成任务**：`.claude/workspace/` 中存在最近 7 天内的未完成 `analyze-*` 目录（缺少 03-conclusion.md）→ 询问用户是否继续
4. **无上下文**：以上都没有 → 请用户提供问题名称（短名称，用于目录命名）和问题描述

确定问题后，创建 Workspace 目录：

```bash
mkdir -p .claude/workspace/analyze-$(date +%Y%m%d)-$NAME
```

## Workspace 归档

Workspace 是产物归档，不是上下文传递管道。

```
.claude/workspace/analyze-YYYYMMDD-$NAME/
├── 01-scope.md            # 问题描述、相关代码、初步假设
├── 02-investigation.md    # [可选] 调查日志：假设验证过程与结果
└── 03-conclusion.md       # 根因、触发机制、修复建议
```

各文件内容指引：
- **01-scope.md**：问题现象、相关日志/报错、涉及的代码模块、初步假设列表
- **02-investigation.md**：每个假设的验证方法、执行结果、排除/确认的证据链（仅在需要多轮验证时生成）
- **03-conclusion.md**：根因定位、完整触发机制、影响范围、修复建议

## 自适应复杂度

根据问题复杂度自动调整流程深度：

- **简单问题**（读代码即可定位）：阶段 1 直接得出结论 → 跳过阶段 2 → 写 01 + 03
- **中等问题**（需要多方验证）：完整三阶段 → 写 01 + 02 + 03
- **复杂问题**（需要实验验证）：完整三阶段 + worktree 实验 → 写 01 + 02 + 03

判断依据：阶段 1 结束时，如果假设明确且证据充分 → 简单；假设多个需要逐一排查 → 中等；需要改代码跑测试才能确认 → 复杂。

## 流程

### 阶段 1：问题定义与初步追踪

1. **理解问题**
   - 从确定的问题来源提取错误现象、触发条件、相关日志
   - 信息不足时向用户澄清，信息充分则自动推进

2. **代码追踪**
   - 阅读相关源代码，追踪调用链和数据流
   - 利用 Grep/Glob 搜索相关模块，利用 Agent (Explore) 进行广度代码探索
   - 标记可疑代码段和潜在问题点

3. **形成假设**
   - 列出可能的根因假设（按可能性排序）
   - 每个假设附带初步证据和待验证点

4. **归档**：写入 `$WS/01-scope.md`

5. **决策**：
   - 假设明确且证据充分 → 直接进入阶段 3 写结论
   - 需要进一步验证 → 进入阶段 2

### 阶段 2：深度调查与实验验证（按需）

对每个待验证假设，选择合适的验证手段：

**代码层面验证**（无需 worktree）：
- 深入阅读更多关联代码
- 追踪边界条件和异常路径
- 比对不同版本的代码变更（git log/diff）

**运行时验证**（无需 worktree）：
- 用 Bash 执行诊断命令（查看进程、端口、日志文件等）
- 运行现有测试套件观察结果
- 请求接口、检查数据库状态等

**实验验证**（需要 worktree）：
- 仅在需要**临时修改代码**才能验证时使用
- 使用 Agent 工具的 `isolation: "worktree"` 参数启动隔离实验
- Agent 在 worktree 中自由修改代码（加调试日志、改参数、写临时测试等）
- Agent 完成后返回实验结果
- 主动清理 worktree：通过 Bash 执行 `git worktree remove <路径> --force` 销毁实验环境

每完成一个假设的验证，追加记录到 `$WS/02-investigation.md`：
```markdown
## 假设 N：<假设描述>
- **验证方法**：<做了什么>
- **结果**：<观察到什么>
- **结论**：确认 / 排除 / 需要更多证据
```

### 阶段 3：根因报告

1. **综合分析**
   - 汇总所有证据，确定根因
   - 描述完整的触发机制（从输入到异常的完整路径）
   - 评估影响范围

2. **修复建议**
   - 给出具体的修复方向和思路
   - 标注需要修改的文件和位置
   - 评估修复复杂度和风险
   - 如果有多种修复方案，列出各方案的优劣

3. **归档**：写入 `$WS/03-conclusion.md`

**⏸ 确认点**：向用户展示根因结论和修复建议
- 确认完成 → 流程结束
- 需要深入 → 补充调查特定方向
- 转入修复 → 建议用户使用 `/f-bugfix` 基于本分析结论进行修复

## 安全保证

- **主分支零影响**：所有代码修改仅在 worktree 隔离环境中发生
- **无 git 写操作**：不执行 git add/commit/push，不产生任何提交记录
- **实验环境即用即毁**：worktree 实验完成后立即销毁，不保留临时分支
- **产出仅为文档**：最终只留下 Workspace 中的分析文档

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：
1. 扫描 `.claude/workspace/analyze-*` 找最近的未完成目录
2. 读取已有的 01/02/03 文件推断进度
3. 向用户确认后从中断点继续

**多阶段项目**：
- 如 `_progress-*.md` 中记录了本任务，流程完成后更新对应子任务状态

## 适用场景

- 生产环境出现异常行为，需要定位根因
- 测试失败但原因不明，需要深入排查
- 性能问题的瓶颈定位
- 理解一段复杂逻辑的执行路径
- 在修复前先搞清楚问题的完整机制

不适合：
- 需要直接修复 Bug → 用 `/f-bugfix`
- 新功能开发 → 用 `/f-dev`
- 纯设计方案 → 用 `/f-design`
- 简单问题直接对话即可解决 → 不需要工作流
