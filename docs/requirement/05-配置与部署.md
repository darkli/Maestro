# 05 - 配置与部署

## 5.1 配置系统

### 5.1.1 配置文件加载顺序

```
1. CLI --config 指定路径（默认值为 config.yaml，即当前目录）
2. ~/.maestro/config.yaml（指定路径不存在时 fallback）
3. 默认值（代码内置，均不存在时使用）
```

### 5.1.2 环境变量替换

配置文件中支持 `${VAR_NAME}` 语法引用环境变量：

```yaml
manager:
  api_key: ${DEEPSEEK_API_KEY}
```

## 5.2 完整配置字段

### 5.2.1 manager - Manager Agent 配置

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `provider` | str | `deepseek` | LLM provider：openai/anthropic/deepseek/gemini/ollama/azure |
| `model` | str | `deepseek-chat` | 模型名称 |
| `api_key` | str | `""` | API Key（建议用 `${ENV_VAR}`） |
| `base_url` | str/null | null | 自定义 API endpoint（OpenAI 兼容） |
| `max_turns` | int | `30` | 单任务最大对话轮数 |
| `max_budget_usd` | float | `5.0` | 单任务费用上限（美元） |
| `request_timeout` | int | `60` | LLM 请求超时（秒） |
| `retry_count` | int | `3` | 失败重试次数（指数退避） |
| `system_prompt` | str | `""` | 内联 system prompt（文件优先） |
| `system_prompt_file` | str | `""` | 主决策 prompt 文件路径 |
| `chat_prompt_file` | str | `""` | /chat 命令 prompt 文件路径 |
| `free_chat_prompt_file` | str | `""` | 自由聊天 prompt 文件路径 |
| `decision_style` | str | `""` | 决策风格：default/conservative/aggressive |

**Provider 自动 base_url**：

| Provider | 自动设置的 base_url |
|----------|-------------------|
| deepseek | `https://api.deepseek.com` |
| ollama | `http://localhost:11434/v1` |
| gemini | `https://generativelanguage.googleapis.com/v1beta/openai/` |

### 5.2.2 coding_tools - 多编码工具配置

```yaml
coding_tools:
  active_tool: claude
  presets:
    claude:
      type: claude
      command: claude
      auto_approve: true
      timeout: 600
    codex:
      type: codex
      command: codex
      auto_approve: true
      sandbox: net-disabled
      model: codex-mini-latest
      skip_git_check: false
      timeout: 600
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `active_tool` | str | `claude` | 当前激活的工具预设名 |
| `presets` | dict | `{}` | 工具预设字典，键为工具名，值为 `CodingToolConfig` |

### 5.2.3 CodingToolConfig 字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `type` | str | `claude` | 工具类型：claude/codex/generic |
| `command` | str | `claude` | 可执行命令名或路径 |
| `extra_args` | list | `[]` | generic 模式的额外参数 |
| `auto_approve` | bool | `true` | Claude: `--dangerously-skip-permissions`；Codex: `--full-auto` |
| `timeout` | int | `600` | 单轮 subprocess 超时（秒） |
| `sandbox` | str | `""` | Codex 沙箱级别：full/net-disabled/off |
| `model` | str | `""` | Codex 模型名 |
| `skip_git_check` | bool | `false` | Codex: `--skip-git-check` |

### 5.2.4 context - 上下文管理配置

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `max_recent_turns` | int | `5` | 保留最近 N 轮对话 |
| `max_result_chars` | int | `3000` | 编码工具输出截断长度（字符） |

### 5.2.5 safety - 安全控制配置

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `max_consecutive_similar` | int | `3` | 死循环检测阈值（连续相同指令次数） |
| `max_parallel_tasks` | int | `3` | 最大并行任务数 |
| `similarity_threshold` | float | `0.85` | 指令语义相似度阈值 |
| `stagnation_threshold` | float | `0.9` | 输出停滞阈值 |
| `max_consecutive_errors` | int | `3` | 连续相同错误上限 |
| `oscillation_window` | int | `6` | action 震荡检测窗口大小 |

### 5.2.6 telegram - Telegram Bot 配置

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | `false` | 是否启用 Telegram Bot |
| `bot_token` | str | `""` | Telegram Bot Token |
| `chat_id` | str | `""` | 授权用户 chat_id（空则不鉴权） |
| `ask_user_timeout` | int | `3600` | ask_user 等待超时（秒） |

### 5.2.7 logging - 日志配置

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `dir` | str | `~/.maestro/logs` | 日志目录 |
| `level` | str | `INFO` | 日志级别：DEBUG/INFO/WARNING/ERROR |
| `max_days` | int | `30` | 日志保留天数 |

## 5.3 配置示例

```yaml
manager:
  provider: deepseek
  model: deepseek-chat
  api_key: ${DEEPSEEK_API_KEY}
  max_turns: 30
  max_budget_usd: 5.0
  system_prompt_file: prompts/system.md
  chat_prompt_file: prompts/chat.md
  free_chat_prompt_file: prompts/free_chat.md
  decision_style: default

coding_tools:
  active_tool: claude
  presets:
    claude:
      type: claude
      command: claude
      auto_approve: true
      timeout: 600
    codex:
      type: codex
      command: codex
      auto_approve: true
      sandbox: net-disabled
      model: codex-mini-latest
      skip_git_check: false
      timeout: 600

context:
  max_recent_turns: 5
  max_result_chars: 3000

safety:
  max_consecutive_similar: 3
  max_parallel_tasks: 3
  similarity_threshold: 0.85
  stagnation_threshold: 0.9
  max_consecutive_errors: 3
  oscillation_window: 6

  enabled: true
  bot_token: ${TELEGRAM_BOT_TOKEN}
  chat_id: ${TELEGRAM_CHAT_ID}
  ask_user_timeout: 3600

  dir: ~/.maestro/logs
  level: INFO
  max_days: 30
```

## 5.4 部署系统（deploy.sh）

### 5.4.1 部署模式

| 模式 | 命令 | 用途 |
|------|------|------|
| 首次部署 | `deploy.sh init` | 完整环境搭建 |
| 增量更新 | `deploy.sh update` | 仅代码和依赖更新 |
| 服务管理 | `deploy.sh service <start\|stop\|restart>` | 管理 daemon |
| 帮助 | `deploy.sh help` | 显示帮助 |
| 交互菜单 | `deploy.sh`（无参数） | 菜单模式 |

### 5.4.2 init 流程（首次完整部署）

#### Phase 1: 文件传输

- 打包项目为 tar.gz（排除 .git / _legacy / .venv 等）
- 通过 rsync 或 git 上传到 VPS

#### Phase 2: 远程环境安装

| 步骤 | 安装内容 |
|------|---------|
| 1 | 系统包：python3, python3-pip, python3-venv, git, curl |
| 2 | 系统级 IPv4 优先：配置 /etc/gai.conf（PREFER_IPV4=true 时） |
| 3 | Python 3.10+（低于 3.10 则安装 3.12） |
| 4 | Node.js 22.x（Claude Code 依赖） |
| 5 | Claude Code：`npm install -g @anthropic-ai/claude-code` |
| 6 | Python venv：`python -m venv .venv && pip install -e .` |
| 7 | 生成 config.yaml（从 deploy.env 变量） |
| 8 | 设置 .env 环境变量 |
| 9 | 创建 systemd 服务（若 Telegram 启用） |

#### Phase 3: 编码工具认证引导

支持 Claude Code 和 Codex CLI 两种编码工具的认证，包含自动备份/恢复机制（详见 §5.4.7）。

| 工具 | 认证文件 | 认证目录 | 登录命令 | 检测方式 |
|------|---------|---------|---------|---------|
| Claude Code | `.credentials.json` | `~/.claude/` | `claude login` | `test -d ~/.claude` |
| Codex CLI | `auth.json` + `config.toml` | `~/.codex/` | `codex login --device-auth` | `test -s ~/.codex/auth.json` |

**认证流程**（每个工具独立执行）：

1. 尝试从本地备份恢复授权文件到服务器（跳过首次部署）
2. 恢复成功后验证有效性（Codex 额外调用 `codex login status`）
3. 若验证通过则跳过手动登录；若失败则清理无效凭据
4. 首次部署或恢复失败时，检查服务器是否已有有效认证
5. 未认证的工具加入 PENDING 列表，统一引导用户在另一终端完成登录
6. 登录成功后自动备份凭据到本地

**Claude Code 特殊处理**：`ANTHROPIC_API_KEY` 非空时跳过整个认证流程（API Key 模式）。

**Codex CLI 特殊处理**：在认证前自动配置 `config.toml` 的 `cli_auth_credentials_store = "file"`，确保 VPS 无浏览器环境下凭据以文件形式持久化。

### 5.4.3 update 流程（增量更新）

**核心特性**：保护用户自定义的 prompts/ 目录。

```
1. 备份远端 prompts/ → /tmp/_maestro_prompts_bak
2. 传输新代码到 VPS
3. 恢复用户自定义 prompts/
4. 系统级 IPv4 优先：配置 /etc/gai.conf（PREFER_IPV4=true 时，幂等）
5. 更新 config.yaml（从 deploy.env 重新生成）
6. 更新 Python 包（pip install -e .）
7. 重启 maestro-daemon（若 Telegram 启用）
```

### 5.4.4 deploy.env 配置

| 参数 | 说明 | 示例 |
|------|------|------|
| `VPS_HOST` | VPS 地址 | `1.2.3.4` |
| `VPS_USER` | SSH 用户 | `root` |
| `VPS_PORT` | SSH 端口 | `22` |
| `VPS_SSH_KEY` | SSH 密钥路径 | `~/.ssh/id_rsa` |
| `VPS_PASSWORD` | SSH 密码（可选） | - |
| `DEPLOY_DIR` | 部署目录 | `/opt/maestro` |
| `DEPLOY_METHOD` | 传输方式 | `rsync` / `git` |
| `MANAGER_PROVIDER` | Manager provider | `deepseek` |
| `MANAGER_MODEL` | Manager 模型 | `deepseek-chat` |
| `MANAGER_API_KEY` | Manager API Key | `sk-xxx` |
| `TELEGRAM_BOT_TOKEN` | Telegram Bot Token | `123456:ABC` |
| `TELEGRAM_CHAT_ID` | 授权 chat_id | `987654321` |
| `CODING_TOOLS` | 编码工具列表（逗号分隔） | `claude,codex` |
| `ANTHROPIC_API_KEY` | Anthropic API Key（Claude API Key 模式） | - |
| `PREFER_IPV4` | 系统级 IPv4 优先（gai.conf） | `true` |
| `MAESTRO_RUN_USER` | 运行用户 | `viber` |
| `SETUP_SYSTEMD` | 是否创建 systemd 服务 | `true` |

### 5.4.5 SSH 连接管理

- **ControlMaster 持久连接**：整个脚本生命周期共用一条 SSH 连接
- 避免重复认证和触发防暴力破解

### 5.4.6 清理卸载（do_clean）

**保留策略**：
- 记录部署前环境快照（`.pre-deploy-state`）
- 仅删除部署时安装的组件，部署前已有的工具保留

**清理项目**：
1. systemd 服务停止 + 移除
2. 部署目录删除
3. bashrc 环境变量清理
4. gai.conf 中 maestro 写入的 IPv4 优先配置
5. ~/.maestro 日志清理
6. ~/.claude 认证删除
7. ~/.codex 认证删除（使用 `MAESTRO_RUN_USER` 的 home 目录）
8. （可选）Claude Code / Node.js 卸载

**注意**：清理仅删除服务器端凭据，本地 `xray/claude-auth-backup/` 备份保留不受影响，下次部署可自动恢复。

### 5.4.7 授权备份/恢复机制

解决 VPS 反复重装部署导致的反复授权问题。首次部署完成授权后，自动将服务器上的凭据文件下载到本地备份；后续重装时自动从本地备份恢复到服务器，免去重复登录。

#### 支持的工具

| 工具 | 备份文件 | 服务器目录 | 本地备份目录 |
|------|---------|-----------|------------|
| Claude Code | `.credentials.json` | `~/.claude/` | `xray/claude-auth-backup/${VPS_HOST}/` |
| Codex CLI | `auth.json`, `config.toml` | `~/.codex/` | `xray/claude-auth-backup/${VPS_HOST}/codex/` |

#### 核心函数

| 函数 | 职责 |
|------|------|
| `_backup_tool_auth(tool, dir, files [, subdir])` | 从服务器下载授权文件到本地备份目录 |
| `_restore_tool_auth(tool, dir, files [, subdir])` | 从本地备份上传授权文件到服务器 |
| `_ensure_codex_file_store(dir)` | 确保服务器上 config.toml 包含 `cli_auth_credentials_store = "file"` |

#### Claude 与 Codex 的控制流差异

| 维度 | Claude Code | Codex CLI |
|------|------------|-----------|
| 认证检测 | `test -d ~/.claude`（通用逻辑） | `test -s ~/.codex/auth.json`（精确检测） |
| 恢复后验证 | 仅检查文件存在 | 调用 `codex login status` 验证 token 有效性 |
| token 过期处理 | 运行时自然发现 | 清理 auth.json + 加入 PENDING 引导重新登录 |
| 预配置 | 无 | `_ensure_codex_file_store` 创建 file 存储模式 |
| case 分支行为 | fall through 到通用 `test -d` | case 内部处理所有逻辑，始终 `continue` |

> Codex 分支不 fall through 到通用 `test -d` 的原因：`_ensure_codex_file_store` 会创建 `~/.codex/` 目录，`test -d` 会产生假阳性（目录存在但无有效凭据）。

#### 场景覆盖

| 场景 | 行为 |
|------|------|
| 首次部署（无备份） | 跳过恢复 → 检测未认证 → PENDING → 用户登录 → 自动备份 |
| 重装部署（token 有效） | 恢复成功 → 验证通过 → 跳过登录 |
| 重装部署（token 过期） | 恢复成功 → 验证失败 → 清理 → PENDING → 重新登录 → 更新备份 |
| 服务器已有授权但本地无备份 | 检测到认证 → 主动下载备份 |
| 卸载后重新部署 | 服务器凭据已删除 → 本地备份保留 → 自动恢复 |
| API Key 模式（Claude） | 跳过整个认证和备份/恢复流程 |

#### 安全性

- 备份文件权限 600，服务器端恢复后权限 600
- `xray/` 目录在 `.gitignore` 中排除
- 日志/终端输出中不打印 token 内容
- SCP 错误输出重定向到 `/dev/null`
- 用户切换使用 `su - $MAESTRO_RUN_USER -c` 确保完整 login shell 环境

#### 可靠性

- 备份/恢复失败不阻塞部署流程（warn 而非 die）
- SCP 失败时清理已上传文件和目录，防止不完整恢复状态
- 恢复后验证失败时清理残留文件，回退到手动登录流程
- 备份文件列表定义为全局变量（`CLAUDE_AUTH_FILES`、`CODEX_AUTH_FILES`），便于维护

## 5.5 目录结构

```
Maestro/
├── pyproject.toml              # 包定义
├── config.example.yaml         # 配置模板

├── prompts/                    # 外置 Prompt 文件
│   ├── system.md               # 主决策 prompt
│   ├── chat.md                 # /chat 命令 prompt
│   └── free_chat.md            # 自由聊天 prompt
├── src/maestro/                # 源代码
│   ├── __init__.py
│   ├── cli.py
│   ├── config.py
│   ├── orchestrator.py
│   ├── tool_runner.py
│   ├── manager_agent.py
│   ├── state.py
│   ├── context.py
│   ├── registry.py
│   └── telegram_bot.py

│       └── ${VPS_HOST}/        # 按 VPS 地址隔离
```

### 5.5.2 运行时目录

```
~/.maestro/
├── config.yaml                 # 用户配置
├── registry.json               # 任务注册表（索引）
├── daemon.pid                  # Telegram Daemon PID
├── logs/                       # 日志目录
│   ├── daemon.log              # Daemon 日志
│   └── tasks/
│       └── {task_id}/          # 每个任务独立目录
│           ├── manager.log     # Manager 决策日志
│           ├── orchestrator.log
│           └── worker.log      # 后台 worker stdout/stderr
└── sessions/
    └── {task_id}/              # 每个任务的 session 目录
        ├── state.json          # 权威状态（所有组件读取此文件）
        ├── checkpoint.json     # 崩溃恢复点
        ├── inbox.txt           # 用户反馈通道
        ├── abort               # 终止信号文件（存在即触发）
        ├── turns.jsonl         # 轮次事件流
        └── report.md           # 最终报告
```
