# 07 - 状态机与协议

## 7.1 任务状态机

### 7.1.1 六个状态

| 状态 | 含义 | 终态 |
|------|------|------|
| `pending` | 已创建，等待启动 | 否 |
| `executing` | 主循环执行中 | 否 |
| `waiting_user` | 等待用户回复 | 否 |
| `completed` | 正常完成 | 是 |
| `failed` | 失败（可恢复） | 是* |
| `aborted` | 用户手动终止 | 是 |

> *代码实现中 `resume` 会直接把状态恢复到 `executing` 并继续主循环。

### 7.1.2 状态转换图

```
                    ┌──────────────────────────────────┐
                    │                                  │
                    ▼                                  │
              ┌──────────┐    start_task()     ┌──────┴───┐
              │ pending  │ ──────────────────→ │ executing │
              └──────────┘                     └─────┬─────┘
                    ▲                                │
                    │ resume()                       │
                    │                    ┌───────────┼───────────┬──────────────┐
              ┌─────┴────┐               │           │           │              │
              │  failed  │ ◄─────────────┤           │           │              │
              └──────────┘  熔断/报错     │           │           │              │
                            超轮/超时     │           │           │              │
                                         │           │           │              │
                                         ▼           ▼           ▼              ▼
                                  ┌────────────┐ ┌────────┐ ┌────────┐ ┌───────────┐
                                  │waiting_user│ │completed│ │ failed │ │  aborted  │
                                  └─────┬──────┘ └────────┘ └────────┘ └───────────┘
                                        │                                     ▲
                                        │ 用户回复                             │
                                        ├─────────→ executing                 │
                                        │                                     │
                                        ├─────────→ failed (超时)              │
                                        └─────────→ aborted ──────────────────┘
```

### 7.1.3 完整转换规则

| 起始状态 | 目标状态 | 触发条件 |
|---------|---------|---------|
| pending → executing | Orchestrator 开始执行 |
| executing → completed | Manager 返回 `done` action |
| executing → failed | 熔断触发 / 运行时异常 / Manager 返回 `blocked` |
| executing → waiting_user | Manager 返回 `ask_user` / 超预算待确认 |
| executing → aborted | 检测到 abort 信号文件 |
| waiting_user → executing | 用户通过 inbox 回复 |
| waiting_user → failed | 等待超时（`ask_user_timeout`） |
| waiting_user → aborted | 检测到 abort 信号 |
| failed → executing | `resume` 命令恢复 |

### 7.1.4 FailReason 枚举

| fail_reason | 触发场景 | 可恢复 |
|-------------|---------|--------|
| `ask_user_timeout` | waiting_user 状态超时未收到回复 | 是 |
| `max_turns` | 当前轮数超过 max_turns | 是 |
| `breaker_tripped` | 死循环检测触发 / 超预算用户拒绝继续 | 是 |
| `blocked` | Manager 判断任务无法继续（返回 blocked action） | 是 |
| `worker_crashed` | Worker 进程异常退出 | 是 |
| `runtime_error` | Python 运行时异常 | 是 |

## 7.2 JSON Action 协议

### 7.2.1 协议概述

Manager Agent 的每次决策**必须**返回一个 JSON 对象，包含以下基础字段：

```json
{
  "action": "<action_type>",
  "instruction": "<给编码工具的指令>",
  "reasoning": "<决策理由>"
}
```

### 7.2.2 五个 Action 类型

#### execute - 执行指令

```json
{
  "action": "execute",
  "instruction": "请在 src/auth.py 中实现 JWT 认证中间件",
  "reasoning": "用户需求中提到了认证功能，首先实现核心中间件"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| action | 是 | 固定为 `"execute"` |
| instruction | 是 | 编码工具可直接执行的具体指令 |
| reasoning | 是 | 决策理由（调试和日志用） |

**后续行为**：ToolRunner 执行 instruction，输出传回 Manager。

---

#### done - 任务完成

```json
{
  "action": "done",
  "instruction": "",
  "reasoning": "所有功能已实现，测试全部通过",
  "summary": "完成了 JWT 登录模块，包含认证中间件、登录接口和单元测试"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| action | 是 | 固定为 `"done"` |
| summary | 是 | 任务最终总结（写入报告） |
| reasoning | 是 | 完成理由 |

**后续行为**：Orchestrator 生成报告，状态变 completed。

---

#### ask_user - 请求用户输入

```json
{
  "action": "ask_user",
  "instruction": "",
  "reasoning": "发现两套鉴权方案，无法自动决定",
  "question": "项目使用 JWT 和 Cookie 两套鉴权，应该全部替换为 Session 吗？"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| action | 是 | 固定为 `"ask_user"` |
| question | 是 | 向用户提出的问题 |
| reasoning | 是 | 为什么需要用户决定 |

**后续行为**：
1. 状态变为 waiting_user
2. Telegram 推送通知（若启用）
3. 等待用户通过 inbox 回复
4. 回复后状态回 executing，反馈合并到下一轮决策
5. 用户回复不会直接变成编码工具指令，而是先交回 Manager 重新决策

---

#### blocked - 无法继续

```json
{
  "action": "blocked",
  "instruction": "",
  "reasoning": "数据库连接失败，缺少 DATABASE_URL 环境变量配置"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| action | 是 | 固定为 `"blocked"` |
| reasoning | 是 | 阻塞原因详细描述 |

**后续行为**：状态变为 failed，fail_reason = blocked。

---

#### retry - 重试上一条

```json
{
  "action": "retry",
  "instruction": "",
  "reasoning": "编码工具返回了临时错误，重试一次"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| action | 是 | 固定为 `"retry"` |
| reasoning | 是 | 重试理由 |

**后续行为**：使用上一轮相同的 instruction 重新执行编码工具。

### 7.2.3 JSON 解析容错

Manager 返回的文本按以下顺序尝试解析：

| 优先级 | 策略 | 示例 |
|--------|------|------|
| 1 | 严格 JSON | `{"action": "execute", ...}` |
| 2 | Code Block 提取 | ` ```json\n{"action": "execute", ...}\n``` ` |
| 3 | 关键词检测 | `##DONE## 任务完成` → done action |
| 4 | 纯文本 Fallback | 整段文字 → execute action 的 instruction |

## 7.3 文件 IPC 协议

### 7.3.1 state.json - 任务权威状态

**位置**：`~/.maestro/sessions/{task_id}/state.json`

**读写方**：

| 操作 | 组件 |
|------|------|
| 写入 | Orchestrator |
| 读取 | CLI（status/list）、Registry（sync）、Telegram Bot（监控） |

**字段**：

```json
{
  "task_id": "a1b2c3d4",
  "status": "executing",
  "requirement": "实现登录模块",
  "working_dir": "/home/user/project",
  "current_turn": 5,
  "max_turns": 30,
  "total_cost_usd": 1.23,
  "tool_cost_usd": 0.80,
  "manager_cost_usd": 0.43,
  "coding_tool_type": "claude",
  "last_instruction": "...",
  "last_output_summary": "...",
  "last_manager_action": "execute",
  "last_manager_reasoning": "...",
  "last_question": "",
  "sub_status": "manager_thinking",
  "fail_reason": null,
  "error_message": null,
  "created_at": "2026-02-27T10:30:00",
  "updated_at": "2026-02-27T10:35:00"
}
```

### 7.3.2 checkpoint.json - 崩溃恢复点

**位置**：`~/.maestro/sessions/{task_id}/checkpoint.json`

**读写方**：

| 操作 | 组件 |
|------|------|
| 写入 | Orchestrator（每轮结束） |
| 读取 | Orchestrator（resume 时） |

**字段**：

```json
{
  "manager_conversation_history": [...],
  "tool_session_id": "claude-session-xxx",
  "current_turn": 5,
  "total_cost_usd": 1.23,
  "manager_cost_usd": 0.43,
  "last_instruction": "...",
  "last_question": "",
  "breaker_state": {
    "instruction_hashes": ["a1b2c3d4", "e5f6g7h8"],
    "current_turn": 5
  }
}
```

### 7.3.3 inbox.txt - 用户反馈通道

**位置**：`~/.maestro/sessions/{task_id}/inbox.txt`

**格式**：每行一条消息

```
2026-02-27T10:30:00|cli|请优先完成登录功能
2026-02-27T10:31:00|telegram|数据库密码是 xxx
```

**字段分隔**：`|`（管道符）

| 字段 | 说明 |
|------|------|
| 时间戳 | ISO 格式 |
| 来源 | cli / telegram |
| 消息内容 | 用户原始文本 |

### 7.3.4 abort - 终止信号文件

**位置**：`~/.maestro/sessions/{task_id}/abort`

**协议**：
- 文件**存在**即表示收到终止信号
- Orchestrator 检测后**删除**文件并退出
- 文件内容无意义（可以为空）

### 7.3.5 turns.jsonl - 轮次事件流

**位置**：`~/.maestro/sessions/{task_id}/turns.jsonl`

**格式**：每行一个 JSON 对象

```json
{"turn": 1, "max_turns": 30, "output_summary": "...", "instruction": "...", "reasoning": "...", "action": "execute", "duration_ms": 1200, "turn_cost": 0.15, "total_cost": 0.42, "timestamp": "..."}
```

**消费者**：Telegram Bot（轮询读取推送 focus 任务详情）

### 7.3.6 registry.json - 任务注册表

**位置**：`~/.maestro/registry.json`

**说明**：索引文件，权威数据在各任务的 state.json 中。Registry 提供快速列表查询和并发检查。

**字段**：

```json
{
  "tasks": {
    "a1b2c3d4": {
      "task_id": "a1b2c3d4",
      "status": "executing",
      "requirement": "实现登录模块",
      "working_dir": "/home/user/project",
      "created_at": "2026-02-27T10:30:00"
    }
  }
}
```

### 7.3.7 report.md - 任务报告

**位置**：`~/.maestro/sessions/{task_id}/report.md`

**生成时机**：任务 completed 时由 Orchestrator 生成。

**内容结构**：

```markdown
# 任务报告

- 任务 ID: a1b2c3d4
- 需求: 实现登录模块
- 状态: completed
- 轮数: 8
- 费用: $1.23
- 耗时: 5 分钟

## 总结

[Manager 的 summary 内容]

## 修改文件

[git diff --name-only 的输出]
```

## 7.4 决策风格协议

### 7.4.1 配置方式

```yaml
manager:
  decision_style: conservative  # default / conservative / aggressive
```

### 7.4.2 风格行为对比

| 决策场景 | default | conservative | aggressive |
|---------|---------|-------------|-----------|
| 不确定的技术选择 | 自行判断 | ask_user | 自行判断 |
| 子任务完成后 | 继续下一步 | ask_user 确认 | 继续下一步 |
| 修改文件前 | 直接修改 | ask_user 确认方案 | 直接修改 |
| 遇到小问题 | 自行解决 | ask_user | 按最佳实践处理 |
| 多个方案可选 | 选最佳 | 列出方案让用户选 | 选最快的 |

### 7.4.3 实现方式

决策风格通过在 system prompt 尾部追加风格说明实现，不改变基础 Prompt 结构。
