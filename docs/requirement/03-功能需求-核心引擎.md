# 03 - 功能需求：核心引擎

## 3.1 Orchestrator - 调度核心

### 3.1.1 职责

管理单个任务的完整执行生命周期：初始化 → 主循环 → 收尾。

### 3.1.2 初始化阶段

| 步骤 | 动作 | 产物 |
|------|------|------|
| 1 | 创建 session 目录 | `~/.maestro/sessions/{task_id}/` |
| 2 | 初始化 ToolRunner | 编码工具配置就绪 |
| 3 | 初始化 ManagerAgent | LLM 客户端 + Prompt 加载 |
| 4 | 初始化 CircuitBreaker | 多维度熔断器就绪 |
| 5 | 初始化 ContextManager | 上下文压缩器就绪 |
| 6 | 创建 inbox.txt | 用户反馈接收通道 |
| 7 | 写入初始 state.json | 状态 = executing |
| 8 | 调用 Manager.start_task() | 生成第一条指令 |

### 3.1.3 主循环（_main_loop）

每轮执行以下步骤：

```
┌─→ 1. 检查 abort 信号
│   2. 读取并清空 inbox（用户反馈）
│   3. 执行编码工具（ToolRunner.run）
│   4. 熔断检查（轮数/工具费用/死循环）
│   5. 合并反馈（如有）调用 Manager 决策
│   6. 解析 JSON Action
│   7. 补充总预算检查（工具费用 + Manager 费用）
│   8. Action 路由：
│      ├─ execute → 更新指令，回到步骤 1
│      ├─ done → 生成报告，退出循环
│      ├─ blocked → 标记失败，退出循环
│      ├─ ask_user → 状态变 waiting_user，等待反馈；用户回复后回到 Manager 重新决策
│      └─ retry → 保持指令不变，回到步骤 1
│   9. 保存 checkpoint
└───────────────────────────────────────┘
```

### 3.1.4 Inbox 管理

#### 写入格式

```
2026-02-27T10:30:00|cli|用户反馈消息
2026-02-27T10:31:00|telegram|来自手机的反馈
```

#### 并发安全

- 写入：fcntl 文件锁（LOCK_EX），保证多进程写入不交错
- 读取：读后立即清空（截断文件），防止重复处理

### 3.1.5 Checkpoint 机制

每轮结束写入 `checkpoint.json`，包含：

| 字段 | 说明 |
|------|------|
| manager_conversation_history | Manager 的完整对话历史 |
| tool_session_id | Claude / Codex 会话 ID（用于 `resume`） |
| current_turn | 当前轮数 |
| total_cost_usd | 工具费用 + Manager 费用的累计总额 |
| manager_cost_usd | Manager LLM 累计费用 |
| last_instruction | 最后一条指令 |
| last_question | 最近一次 ask_user 的问题文本 |
| breaker_state | 熔断器内部状态（轮数、hash、错误指纹等） |

### 3.1.6 报告生成

任务完成时生成 `report.md`，内容：

- 任务基本信息（ID、需求、状态、工具类型）
- 执行统计（轮数、耗时、费用）
- Manager 总结（summary 字段）
- 修改文件列表（通过 `git diff` 获取）

### 3.1.7 事件流

`turns.jsonl`：每轮写入一行 JSON，供 Telegram Daemon 读取推送。

## 3.2 ManagerAgent - 外层决策 AI

### 3.2.1 职责

分析编码工具输出，决定下一步动作。以 JSON 格式返回结构化决策。

### 3.2.2 Provider 支持

| Provider | SDK | base_url | 说明 |
|----------|-----|----------|------|
| openai | openai | 自定义或默认 | 原生 OpenAI |
| deepseek | openai | `https://api.deepseek.com` | 自动设置 |
| gemini | openai | `https://generativelanguage.googleapis.com/v1beta/openai/` | 自动设置 |
| ollama | openai | `http://localhost:11434/v1` | 本地模型 |
| azure | openai | 自定义 | Azure OpenAI |
| anthropic | anthropic | - | 原生 Anthropic SDK |

### 3.2.3 决策方法

#### start_task(requirement)

- 初始化对话历史
- 发送需求描述
- 返回第一条 instruction

#### decide(tool_output)

- 将编码工具输出加入对话历史
- 调用 LLM 获取 JSON 决策
- 解析并返回 Action 对象

#### decide_with_feedback(tool_output, feedback)

- 将用户反馈合并到 user message 中
- 其余同 decide()

#### standalone_chat(task_context, question)

- **独立会话**：不修改主循环对话历史
- 使用 chat_prompt_file 的 system prompt
- 返回自然语言回复

#### free_chat(message)

- Telegram 自由聊天
- 维护独立的多轮历史（最多 20 条）
- 使用 free_chat_prompt_file
- `/new` 命令可重置历史

### 3.2.4 JSON 解析策略

尝试顺序（从严到宽）：

1. **严格 JSON**：直接 `json.loads(response)`
2. **Code Block 提取**：匹配 ` ```json {...} ``` ` 后解析
3. **关键词检测**：`##DONE##` → done action，`##BLOCKED##` → blocked action
4. **纯文本 Fallback**：整个回复视为 execute 的 instruction

### 3.2.5 费用估算

内置 20+ 模型的 token 定价表（`MODEL_PRICING` 字典），根据 prompt_tokens 和 completion_tokens 估算费用。未知模型 fallback 到 0 费用。

### 3.2.6 重试机制

- 重试次数：`manager.retry_count`（默认 3）
- 退避策略：指数退避（1s, 2s, 4s）
- 重试场景：网络错误、超时

## 3.3 ToolRunner - 编码工具运行器

### 3.3.1 职责

调用编码工具执行指令，解析输出结果。

### 3.3.2 三种运行模式

#### Claude 模式（`type: claude`）

**启动命令**：

```bash
claude -p --output-format json [--dangerously-skip-permissions] [--resume <session_id>] "<instruction>"
```

| 参数 | 条件 | 说明 |
|------|------|------|
| `-p` | 始终 | prompt 模式（非交互） |
| `--output-format json` | 始终 | JSON 格式输出 |
| `--dangerously-skip-permissions` | `auto_approve: true` | 跳过权限确认 |
| `--resume <session_id>` | 非首轮 | 恢复上一轮会话上下文 |

**输出解析**：

```json
{
  "session_id": "abc123",
  "result": "代码修改完成...",
  "cost_usd": 0.15,
  "is_error": false,
  "subtype": "success"
}
```

**Fallback**：JSON 解析失败时降级为纯文本模式。

#### Generic 模式（`type: generic`）

**启动命令**：

```bash
<command> [extra_args...] "<instruction>"
```

**输出**：stdout + stderr 合并为纯文本。

**限制**：
- 不支持费用追踪
- 不支持会话恢复（无 session_id）
- 不区分 JSON / 纯文本输出

#### Codex 模式（`type: codex`）

**启动命令**：

```bash
codex exec --json [--full-auto] [--sandbox <level>] [--model <model>] [--skip-git-check] [resume <session_id>] "<instruction>"
```

| 参数 | 条件 | 说明 |
|------|------|------|
| `exec --json` | 始终 | JSONL 非交互模式 |
| `--full-auto` | `auto_approve: true` | 自动批准 |
| `--sandbox <level>` | `sandbox` 非空 | 沙箱级别：full / net-disabled / off |
| `--model <model>` | `model` 非空 | Codex 模型名 |
| `--skip-git-check` | `skip_git_check: true` | 跳过 git 仓库检查 |
| `resume <session_id>` | 非首轮 | 恢复上一轮 Codex 会话 |

**输出解析**：

- `thread.started`：提取会话 ID
- `item.completed(type=agent_message)`：提取工具输出文本
- `turn.completed.usage`：提取 token 用量并估算费用
- `turn.failed`：标记错误并提取错误信息

### 3.3.3 RunResult 数据结构

| 字段 | 类型 | 说明 |
|------|------|------|
| output | str | 编码工具输出文本 |
| session_id | str/None | Claude / Codex 会话 ID |
| cost_usd | float | 本轮费用（generic 模式为 0） |
| duration_ms | int | 执行时间（毫秒） |
| is_error | bool | 是否出错 |
| error_type | str/None | 错误类型 |

### 3.3.4 超时处理

- 配置：`coding_tool.timeout`（默认 600 秒）
- 超时后：subprocess 被 kill，返回超时错误消息
- Manager 通常会 retry 或调整指令

## 3.4 PromptLoader - Prompt 热加载

### 3.4.1 职责

从外部 Markdown 文件加载 Prompt，支持热加载（修改后自动生效）。

### 3.4.2 加载逻辑

```
1. 检查文件是否存在
   ├─ 不存在 → 自动生成默认内容并写入
   └─ 存在 → 检查 mtime 是否变更
              ├─ 未变 → 返回缓存
              └─ 已变 → 重新读取并更新缓存
```

### 3.4.3 支持的 Prompt 文件

| 配置字段 | 文件路径 | 用途 |
|----------|---------|------|
| `system_prompt_file` | prompts/system.md | Manager 决策 system prompt |
| `chat_prompt_file` | prompts/chat.md | /chat 命令问答 |
| `free_chat_prompt_file` | prompts/free_chat.md | Telegram 自由聊天 |

### 3.4.4 优先级

`system_prompt_file` 优先于内联的 `system_prompt` 字符串。

### 3.4.5 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 文件不存在 | 自动生成默认内容 |
| 目录不存在 | 自动创建目录 |
| 编码错误 | Fallback 到默认值 |
| 权限不足 | Fallback 到默认值 |
| 空文件 | 使用默认内容 |
| 文件被删除 | 下次调用重新生成 |

## 3.5 ContextManager - 上下文压缩

### 3.5.1 职责

控制 Manager 对话历史长度，降低 token 成本。

### 3.5.2 压缩策略（build_context）

| 对话长度 | 策略 |
|---------|------|
| ≤ `max_recent_turns * 2` 条消息 | 全量传递 |
| > `max_recent_turns * 2` 条消息 | 首轮 + 最近 N 轮 + 中间摘要 |

**接入位置**：
- Manager 内部维护完整 `conversation_history`
- 真正调用 LLM 前，先用 `ContextManager.build_context()` 压缩后再发送

### 3.5.3 输出截断（truncate_output）

当编码工具输出超过 `max_result_chars`（默认 3000 字符）时：

- 保留前 1/3 内容（通常是命令开始部分）
- 保留后 2/3 内容（通常包含执行结果/错误信息）
- 中间插入 `...（已截断）...` 标记

**设计原因**：尾部通常包含执行结果和错误信息，对 Manager 决策更关键。

## 3.6 CircuitBreaker - 熔断器

### 3.6.1 多维度熔断

#### 轮数限制

```
当前轮数 > max_turns → 熔断
```

- 默认阈值：30 轮
- 配置项：`manager.max_turns`

#### 费用限制

```
任务总费用（工具费用 + Manager 费用）> max_budget_usd → 熔断
```

- 默认阈值：$5.0
- 配置项：`manager.max_budget_usd`
- **实现细节**：
  - CircuitBreaker 内部先检查工具费用
  - Manager 决策完成后由 Orchestrator 补做总费用检查，避免遗漏 Manager 成本
- **特殊处理**：超预算时不立即失败，先通知用户确认。用户确认后预算翻倍继续当前任务。

#### 死循环检测

```
连续 N 次指令的 MD5 hash 相同 → 熔断
```

- 默认阈值：3 次
- 配置项：`safety.max_consecutive_similar`
- hash 算法：`md5(instruction).hexdigest()[:8]`

#### 其他检测维度

- 指令语义相似度：`safety.similarity_threshold`
- 相同错误重复：`safety.max_consecutive_errors`
- 输出停滞：`safety.stagnation_threshold`
- action 震荡：`safety.oscillation_window`
