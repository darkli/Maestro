# 06 - 非功能需求

## 6.1 安全性

### 6.1.1 进程安全

| 机制 | 实现 | 保护目标 |
|------|------|---------|
| 文件锁 | fcntl LOCK_EX | inbox.txt 多进程写入不交错 |
| 原子写入 | 临时文件 + os.rename | state.json / checkpoint.json 不出现半写 |
| 信号文件 | abort 文件存在即触发 | 进程间终止通信 |

### 6.1.2 数据安全

| 机制 | 说明 |
|------|------|
| API Key 环境变量化 | 配置文件中用 `${VAR}` 引用，不硬编码 |
| Telegram 鉴权 | 仅授权 chat_id 可操作 |
| 密码安全传输 | deploy.sh 中通过管道传 chpasswd，不走 shell 参数 |
| 授权凭据备份安全 | 本地/服务器端权限 600；`xray/` 在 `.gitignore` 排除；日志不打印 token 内容；SCP 错误输出 `/dev/null` |

### 6.1.3 熔断保护

| 维度 | 默认阈值 | 目的 |
|------|---------|------|
| 轮数 | 30 轮 | 防止无限循环消耗资源 |
| 费用 | $5.0 | 防止任务总费用（工具 + Manager）失控 |
| 死循环 | 3 次相同指令 | 检测 Manager 陷入重复 |
| 语义相似/错误重复/输出停滞/action 震荡 | safety 配置值 | 识别更隐蔽的卡死或空转 |

### 6.1.4 并发控制

- 最大并行任务数：`safety.max_parallel_tasks`（默认 3）
- 检查范围：pending + executing + waiting_user 状态
- 超限时拒绝新任务，给出明确提示

## 6.2 可靠性

### 6.2.1 崩溃恢复

| 场景 | 恢复方式 |
|------|---------|
| Worker 进程崩溃 | `maestro resume <task_id>` 从 checkpoint 恢复 |
| SSH 断开 | nohup 进程保活，重连后通过 `maestro status` 查看进度 |
| 编码工具超时 | ToolRunner 返回超时错误，Manager 决策重试或调整 |
| LLM API 网络错误 | 指数退避重试（1s → 2s → 4s） |

### 6.2.2 数据完整性

| 操作 | 保护措施 |
|------|---------|
| state.json 写入 | atomic_write_json（临时文件 + rename） |
| registry.json 写入 | atomic_write_json |
| checkpoint.json 写入 | atomic_write_json |
| inbox.txt 写入 | fcntl 文件锁 |
| registry 损坏 | rebuild() 从所有 state.json 重建 |

### 6.2.3 降级策略

| 组件 | 降级行为 |
|------|---------|
| JSON 解析失败 | Fallback 到纯文本模式（视为 execute） |
| Prompt 文件损坏 | Fallback 到内置默认 Prompt |
| 费用估算未知模型 | Fallback 到 0 费用 |
| 授权备份/恢复失败 | warn 不阻塞，回退到手动登录流程 |
| 授权 token 过期 | 清理无效凭据，引导用户重新登录，登录后更新备份 |

## 6.3 性能

### 6.3.1 上下文压缩

- 长对话历史自动压缩（首轮 + 最近 N 轮 + 摘要）
- 编码工具输出超 3000 字符自动截断
- 目的：降低 LLM token 成本，防止上下文超长

### 6.3.2 Prompt 缓存

- PromptLoader 基于文件 mtime 缓存
- 文件未变更时直接返回缓存，避免频繁磁盘 IO

### 6.3.3 SSH 连接复用

- deploy.sh 使用 ControlMaster 持久连接
- 整个部署过程共用一条 SSH 连接

## 6.4 可扩展性

### 6.4.1 多 Provider 架构

新增 LLM Provider 的步骤：

1. 在 `manager_agent.py` 的 Provider 工厂中添加分支
2. 如使用 OpenAI 兼容协议，仅需配置 base_url
3. 如使用原生 SDK，需添加新的客户端初始化逻辑

### 6.4.2 编码工具可插拔

新增编码工具的步骤：

1. 在 `coding_tools.presets` 中新增一个工具预设
2. 预设中的 `type` 设为 `generic`（或实现专用分支如 `codex`）
3. 将 `coding_tools.active_tool` 切到新预设或通过 `maestro switch` 切换
4. `command/extra_args` 传递工具特有参数

适配的工具示例：Gemini CLI、Aider、Continue 等。

### 6.4.3 Prompt 外置化

- 所有 Prompt 为独立 Markdown 文件
- 修改后无需重启服务（热加载）
- 支持不同项目使用不同 Prompt

### 6.4.4 决策风格

通过 `decision_style` 配置切换 Manager 行为：

| 风格 | 特点 |
|------|------|
| `default` | 平衡模式，小问题自行决定，重大决策问用户 |
| `conservative` | 保守模式，频繁确认，逐步推进 |
| `aggressive` | 激进模式，尽量自行决定，快速完成 |

## 6.5 可维护性

### 6.5.1 代码规范

| 规范 | 约定 |
|------|------|
| 缩进 | 4 空格 |
| 文件命名 | snake_case.py |
| 类名 | PascalCase |
| 函数/变量 | snake_case |
| 配置建模 | dataclass |
| 注释语言 | 中文 |
| 提交消息 | 中文，动作导向 |

### 6.5.2 模块职责单一

| 模块 | 单一职责 |
|------|---------|
| cli.py | 仅负责命令解析和路由 |
| orchestrator.py | 仅负责任务生命周期调度 |
| manager_agent.py | 仅负责 LLM 调用和决策解析 |
| tool_runner.py | 仅负责编码工具调用 |
| state.py | 仅负责状态管理和安全工具 |

### 6.5.3 日志体系

| 日志文件 | 内容 | 消费者 |
|---------|------|--------|
| manager.log | Manager 决策过程 | 开发者排查 |
| worker.log | 后台 worker 的 stdout/stderr 汇总 | 开发者排查 |
| orchestrator.log | 调度流程日志 | 开发者排查 |
| turns.jsonl | 结构化轮次事件 | Telegram 推送 |
| daemon.log | Telegram Daemon 日志 | 运维排查 |

## 6.6 兼容性

### 6.6.1 Python 版本

- 最低要求：Python 3.10+
- deploy.sh 在低于 3.10 时自动安装 Python 3.12

### 6.6.2 操作系统

| 系统 | 支持情况 |
|------|---------|
| Linux (x86_64) | 完全支持 |
| Linux (aarch64) | 完全支持 |
| macOS | 支持 |
| Windows | 不支持（依赖 fcntl / nohup 等 Unix 工具） |
