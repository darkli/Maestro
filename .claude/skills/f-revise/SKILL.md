# Review 反馈研判 Skill

本 Skill 是一份流程指引——收到代码 review 反馈后，先独立研判每条意见的合理性，对有分歧的条目先反馈给用户讨论，达成共识后再修改代码。**在用户确认前不修改任何代码。**

## 入口逻辑

按以下优先级确定 review 内容来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-revise` 后粘贴了 review 内容 → 直接使用
2. **对话上下文**：当前对话中已包含 review 反馈内容 → 提取要点，向用户确认是否完整
3. **无内容**：以上都没有 → 请用户提供 review 内容

## 流程

### 阶段 1：研判分析

对 review 中的**每条意见**，执行以下步骤：

1. **理解意见**：准确理解 reviewer 的建议及其意图
2. **查看现状**：阅读相关代码，理解当前实现的上下文和设计意图
3. **独立判断**：基于代码现状，对该条意见给出判断：
   - ✅ **同意**：reviewer 说得对，应该改
   - ⚠️ **部分同意**：方向对但具体做法有更好的方案
   - ❌ **不同意**：当前实现有合理理由，不应改动

每条意见的判断必须附带**具体理由**（引用代码、说明上下文、解释权衡），不能只说"同意/不同意"。

分析完成后，输出研判结果汇总表：

```
| # | Review 意见摘要 | 判断 | 理由 |
|---|----------------|------|------|
| 1 | ...            | ✅   | ...  |
| 2 | ...            | ❌   | ...  |
```

**⏸ 确认点**：

- **无分歧**（全部同意）→ 直接列出修改清单并进入阶段 2 实施
- **有分歧**（存在部分同意或不同意）→ 逐条展示分歧理由，等待用户裁定
  - 用户可对每条分歧做出裁定：按 review 改 / 按当前判断保留 / 用户给出第三方案
  - 所有分歧裁定完毕后，进入阶段 2

### 阶段 2：实施修订

1. **确定修改清单**：汇总所有需要修改的条目（同意的 + 用户裁定要改的）
2. **逐条实施**：按修改清单逐条修改代码
   - 遵守项目的 i18n、design-system、code style 等规范
   - 每条修改完成后简要说明改了什么
3. **自查**：
   - 运行相关测试确认无回归
   - 修改是否引入新问题
   - 是否需要更新测试
   - 是否需要更新 i18n 文件
4. **汇报**：向用户报告所有修改完成

## 自适应复杂度

- **简单 review**（1-3 条意见，无争议）：快速走完两个阶段，不需要详细表格
- **常规 review**（多条意见，有部分分歧）：完整流程，表格展示
- **大规模 review**（大量意见涉及多文件）：按文件或模块分组分析，分批实施

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：review 研判通常在单次对话内完成，不需要跨对话恢复。如果中断，用户重新提供 review 内容即可。

## 适用场景

- 收到另一个 Claude Code 实例的 review 反馈
- 收到人工 code review 意见
- 收到 PR review comments 需要逐条处理
- 任何需要"先判断再修改"的反馈处理场景

不适合：
- 自己发起 code review → 直接对话或用 Agent
- 新功能开发 → 用 `/f-dev`
- Bug 修复 → 用 `/f-bugfix`
