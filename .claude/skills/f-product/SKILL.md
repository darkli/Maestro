# 产品设计 Skill

本 Skill 是一份流程指引——从产品经理视角定义"做什么"和"为什么做"，不写代码、不做技术设计。适合从模糊想法出发、需要先理清产品方案再进入技术设计的场景。

## 入口逻辑

按以下优先级确定需求来源（取第一个命中的）：

1. **用户输入**：用户在 `/f-product` 后提供了产品想法 → 直接使用
2. **对话上下文**：当前对话中已讨论过产品想法或用户需求 → 提取要点，向用户确认理解是否准确
3. **未完成任务**：`.claude/workspace/` 中存在最近 7 天内的未完成 `product-*` 目录（缺少 03-spec.md）→ 询问用户是否继续
4. **无上下文**：以上都没有 → 请用户提供产品名称和想法描述

确定想法后，创建 Workspace 目录：

```bash
mkdir -p .claude/workspace/product-$(date +%Y%m%d)-$NAME
```

## Workspace 归档

Workspace 是产物归档，不是上下文传递管道。

```
.claude/workspace/product-YYYYMMDD-$NAME/
├── 01-problem.md     # 问题定义
├── 02-solution.md    # 方案设计
└── 03-spec.md        # 功能规格（PRD）
```

各文件内容指引：
- **01-problem.md**：核心问题、目标用户画像、使用场景、成功指标、现有能力分析
- **02-solution.md**：交互流程/功能流程、信息架构、状态矩阵、关键设计决策
- **03-spec.md**：功能清单（MoSCoW 优先级）、MVP 范围、验收标准、依赖关系、风险与假设

## 流程

### 阶段 1：问题定义

1. **提炼问题**
   - 从用户想法中区分"问题"和"解决方案"
   - 明确核心问题是什么

2. **识别用户**
   - 目标用户画像
   - 典型使用场景
   - 可衡量的成功指标

3. **分析现状**
   - 梳理项目已有的可复用能力
   - 识别约束条件

4. **归档**：写入 `$WS/01-problem.md`

**⏸ 确认点**：向用户展示问题定义，获得确认
- 确认通过 → 进入阶段 2
- 要求修改 → 调整后重新确认
- 终止 → 保留已有产物

### 阶段 2：方案设计

1. **设计方案**
   - 根据项目类型（前端/后端/全栈）设计对应维度：
     - 前端：信息架构、页面线框（ASCII）、交互流程（Mermaid）、状态矩阵
     - API 服务：API 体验设计、调用流程（序列图）、错误码体系
   - 如 CLAUDE.md 配置了 design-system → 线框图标注设计令牌
   - 如 CLAUDE.md 配置了 i18n → 考虑多语言文案长度差异

2. **记录决策**
   - 关键设计决策及理由

3. **归档**：写入 `$WS/02-solution.md`

**⏸ 确认点**：向用户展示方案设计，获得确认
- 确认通过 → 进入阶段 3
- 要求修改 → 调整后重新确认

### 阶段 3：功能规格（PRD）

1. **提取功能点**
   - 从方案中提取所有功能点
   - 按 MoSCoW 标注优先级

2. **划定 MVP**
   - 确定 MVP 范围
   - 为每个 Must Have 功能定义验收标准
   - 梳理功能间的依赖关系

3. **识别风险**
   - 技术风险和产品假设

4. **归档**：写入 `$WS/03-spec.md`

**⏸ 确认点**：向用户展示 PRD，获得确认
- 确认完成 → 流程结束
- 要求修改 → 调整后重新确认

## 完成与后续衔接

产品设计完成后，后续可使用：
- `/f-design` — 技术设计（会自动检测到 `product-*` 上游）
- `/f-dev` — 直接开发

## 恢复机制

**同一对话**：对话上下文自然保持，直接继续。

**跨对话**：
1. 扫描 `.claude/workspace/product-*` 找最近的未完成目录
2. 读取已有的 01/02/03 文件推断进度
3. 向用户确认后从中断点继续

**多阶段项目**：
- 如 `_progress-*.md` 中记录了本任务，流程完成后更新对应子任务状态
- 用户说"继续做"时，检查 progress 文件确定下一步

## 适用场景

- 模糊想法具体化（如"我想做一个监控面板"）
- 新功能的产品定义（先定义再设计再开发）
- 产品方案评审（只产出 PRD 不写代码）
- MVP 范围划定（如"这个功能先做哪些"）
- 复杂功能的前期拆解（如"用户权限系统怎么定义"）

不适合：
- 已有明确需求直接做技术设计 → 用 `/f-design`
- 已有设计方案直接写代码 → 用 `/f-dev`
- Bug 修复 → 用 `/f-bugfix`
