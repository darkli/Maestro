# 智能测试 Skill

本 Skill 是一份流程指引——无需参数，自动分析当前变更上下文，完成从测试评估到覆盖率验证的完整流程。由 Claude 在主对话中直接执行，通过读取 CLAUDE.md 的 Testing 部分获取项目测试规范。

## 前置检查

读取 CLAUDE.md 的 Capabilities 节，检查 `testing` 标签值：

- `testing: false` → 提示"测试框架未配置"，建议配置后运行 `/f-init -u`，然后退出
- 非 false → 正常执行

## 流程

### 阶段 1：上下文分析

1. **识别变更范围**（按优先级，多个来源合并去重）
   - 对话上下文：是否刚完成 `/f-dev` 或 `/f-bugfix`，提取涉及的文件
   - 未提交变更：通过 `git diff` 和 `git status` 获取
   - 最近提交：通过 `git log --name-only -1` 检查最近一次 commit 的变更文件（未 push 的本地提交）
   - 以上均无 → 询问用户要测试哪些模块

2. **定位相关测试**
   - 对每个变更的源文件，查找对应的测试文件（按 CLAUDE.md 约定的文件放置规则）
   - 标记：有测试文件 / 无测试文件

### 阶段 2：测试评估与设计

1. **评估现有测试**（对已有测试文件的源文件）
   - 阅读源代码和测试代码
   - 对比变更涉及的函数/分支与已有 test case，识别覆盖缺口
   - 识别缺失的测试场景：新增分支、边界条件、错误路径

2. **设计补充测试**（对无测试文件或测试不完善的源文件）
   - 分析源代码：函数签名、分支逻辑、边界条件、依赖关系和 Mock 需求
   - 列出需要新增的测试用例

3. **展示测试计划**

   **⏸ 确认点**：向用户展示评估结果和补充计划
   - 哪些文件测试已完善（无需改动）
   - 哪些文件需要补充哪些用例
   - Mock 策略
   - 确认通过 → 进入阶段 3
   - 调整 → 修改计划后继续
   - 测试已完善无需补充 → 直接跳到阶段 4

### 阶段 3：编写测试

1. **实现测试用例**
   - 遵循 CLAUDE.md Testing 部分的约定（环境、import 方式、文件放置）
   - AAA 模式（Arrange-Act-Assert）
   - 描述性测试名称（`should xxx when yyy`）
   - 已有测试文件 → 追加；无 → 创建新文件

### 阶段 4：运行与验证

1. **运行测试**
   - 执行完整测试套件（确保新测试通过且不破坏现有测试）
   - 如有失败 → 分析原因，修复测试代码（最多 3 轮）
   - 展示结果：通过 / 失败 / 跳过 / 耗时

2. **覆盖率分析**（如项目支持）
   - 尝试运行覆盖率测试
   - 如覆盖率工具未配置或命令失败 → 跳过，仅提示"覆盖率未配置"
   - 正常输出时：展示变更文件的覆盖率（statements / branches / functions / lines）
   - 如有新增测试，对比覆盖率提升

3. **输出摘要**

   **⏸ 确认点**：向用户展示测试总结
   - 变更文件清单及对应测试文件
   - 新增测试用例数
   - 测试结果（全部通过 / 有失败）
   - 覆盖率数据（如有）
   - 确认完成 → 流程结束
   - 要求修改 → 调整后重新运行

## 复杂度自适应

不需要用户选择模式，Claude 根据变更规模自行调整：

- **小变更**（1-2 个文件、逻辑简单）：阶段 2 简化为快速评估，确认点精简展示
- **中等变更**（跨几个文件）：完整四阶段
- **大变更**（跨多模块、新增 API）：阶段 2 深入分析每个模块的测试缺口，阶段 3 分批编写
